# Schema



## Structured schema tables

The following tables organize the schema into clear column definitions. Types are proposed based on the notes; adjust as needed during implementation.

### user_profiles

| Column        | Type        | Description |
|---------------|-------------|-------------|
| id            | uuid        | PK |
| first_name    | text        |  |
| last_name     | text        |  |
| suffix        | text        | e.g., Jr., III |
| title         | text        | e.g., Dr., Engr., Atty. |
| phone_number  | text        |  |
| email         | text        |  |
| birth_date    | date        |  |
| photo_url     | text        | Higher resolution photo URL |
| thumb_url     | text        | Thumbnail URL (could be auto-generated on upload) |
| active        | boolean     | Whether the profile is active, this will  be determined by org partners |

### professionals

| Column           | Type             | Description |
|------------------|------------------|-------------|
| user_profile_id  | uuid             | PK, references `user_profiles.id` |
| profile_summary  | text             | Professional summary |
| tags             | text[]           | May be auto-generated by AI |
| value_profile    | text             | Concatenated summary of education, work history, and skills |
| embedding_vector | vector           | Vectorized representation of `value_profile` (pgvector). ADD THIS LATER WHEN WE GONNA RAG |
| updated_at       | timestamptz      | Last update timestamp |

> Note: `value_profile` should be auto-updated on insert/update trigger and used for vectorization.

### education

| Column          | Type        | Description |
|-----------------|-------------|-------------|
| id              | uuid        | PK |
| professional_id | uuid        | FK to `professionals(uid)` |
| school          | text        |  |
| degree          | text        | Can include courses, short courses, etc. |
| year            | int         |  |
| created_at      | timestamptz |  |
| updated_at      | timestamptz |  |

### work_history

| Column          | Type        | Description |
|-----------------|-------------|-------------|
| id              | uuid        | PK |
| professional_id | uuid        | FK to `professionals(uid)` |
| role            | text        |  |
| organization    | text        |  |
| summary         | text        |  |
| year_start      | int         |  |
| year_end        | int         | Nullable if still active |
| created_at      | timestamptz |  |
| updated_at      | timestamptz |  |

### skills

| Column          | Type               | Description |
|-----------------|--------------------|-------------|
| id              | uuid               | PK |
| professional_id | uuid               | FK to `professionals(uid)` |
| skill           | text               |  |
| level           | skill_level (enum) | `beginner` | `intermediate` | `expert` |
| created_at      | timestamptz        |  |

> Consider creating the `skill_level` enum and appropriate FKs/indexes for efficient querying.

### links

| Column          | Type        | Description |
|-----------------|-------------|-------------|
| id              | uuid        | PK |
| professional_id | uuid        | FK to `professionals(uid)` |
| url             | text        |  |
| summary         | text        | hopefully this will be auto-generated by AI after scrapping |
| created_at      | timestamptz |  |
| updated_at      | timestamptz |  |
